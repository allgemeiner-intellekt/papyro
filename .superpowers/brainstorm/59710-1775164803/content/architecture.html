M2 Import Pipeline — Architecture

Service decomposition and data flow

Import Pipeline Architecture
User drops PDF(s) onto window
↓
ImportCoordinator
Orchestrates the pipeline. Creates a Paper model with status "importing", drives each step, updates state on completion or failure.
STEP 1
FileService
Copy PDF to papers/ with temp UUID name
STEP 2
TextExtractor
PDFKit: extract text from first 5 pages → save to .cache/text/
STEP 3
IdentifierParser
Regex scan for DOI, arXiv ID, PMID, ISBN from extracted text
STEP 4
MetadataService
Protocol → TranslationServerProvider (primary) or CrossRefProvider (fallback)
STEP 5
IndexService
Write per-paper JSON to index/, regenerate _all.json
STEP 6
FileService
Rename PDF to 2024_chen_attention.pdf
↓
✓ Resolved
Metadata found, PDF renamed, index written
⚠ Unresolved
No metadata found — PDF kept with temp name, retry later

Key Design Decisions

  • papers/ is flat — no year subfolders. All organization via symlinks.
  • Copy only — no move option. Original PDF untouched.
  • MetadataService is a protocol — real (translation-server) and mock implementations.
  • Async pipeline — each step is async, UI updates reactively via @Observable Paper model.
  • Rename on resolve — PDF starts as UUID, renamed to year_author_title.pdf when metadata arrives.
